# `DD-384` Harden the managed verbose-trace progress callback (E2E finding)

## Method

`TDD + BDD` (reproduce in a real `dashboard telegram-claude.check-message` run, then add coverage)

## Problem

During the live noVNC E2E for `DD-383` (2026-06-11), a real inbound Telegram
message drove the bridge to resume the real Claude session and deliver the
reply successfully (`reply.sent`). However, the managed verbose-trace
`on_progress` callback raised `Claude progress callback failed` on every
streamed stream-json line (`Session resumed`, `Agent: …`, `Turn completed`).

The skill's design correctly treats these as **non-fatal** — the final reply
was still generated and delivered — so this did not break the round-trip. But
the in-chat streaming verbose trace did not update cleanly against the real
Telegram API in this environment, which the mocked unit tests do not surface.

## Acceptance

- reproduce the progress-callback failure in a real (or faithfully simulated)
  `check-message` run and capture the underlying error (the callback currently
  swallows the detail into a generic `Claude progress callback failed` audit
  line — surface the real cause: e.g. `editMessageText` failure, message-not-
  found, rate limit, or empty trace message)
- root-cause why the streamed verbose-trace edits fail while the initial
  kickoff trace and the final `reply.sent` succeed
- fix so the verbose trace updates cleanly, or degrades to a documented,
  intentional fallback rather than a per-line failure
- add unit coverage for the real failure mode and the fixed path
- Docker tests pass; `lib/Telegram/Claude/Manager.pm` stays at `100%`
- `tickets/TESTING.md`, `docs/`, `Changes` updated; release commit and push
