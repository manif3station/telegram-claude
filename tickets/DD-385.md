# `DD-385` Fix live-pane discovery: tmux mangles tab separators in `-F` output

## Method

`TDD + BDD` (root-caused live in the noVNC lab, then fixed with coverage)

## Problem

During the live noVNC E2E (proving §6.2 live-TUI injection and §7 TUI→Telegram
mirroring), the bridge never discovered the live `claude --resume <session-id>`
tmux pane, so it could never inject a Telegram turn into the live TUI or mirror
TUI-originated turns back to Telegram — it always fell back to detached resume.

Root cause: `tmux_pane_rows` ran
`tmux list-panes -a -F '#{pane_id}<TAB>#{pane_tty}<TAB>#{pane_current_command}'`
and split the output on `\t`. But **tmux rewrites a literal tab in a `-F`
format to an underscore in its output** (confirmed by hexdump: the separator
byte is `0x5f`, not `0x09`). So every pane collapsed into a single field with
`tty` undefined, the `next if !defined $tty` dropped it, and `tmux_pane_rows`
returned zero panes → `resolve_claude_live_tmux_pane` returned `(none)`.

The unit suite mocks `tmux_panes_runner` (and the one real-branch test used a
fake `tmux` that emitted real tabs), so the real-tmux tab-mangling was never
exercised.

## Fix

Use a pipe `|` separator, which tmux preserves and which never appears in a
pane id (`%N`), tty (`/dev/pts/N`), or current command:
`-F '#{pane_id}|#{pane_tty}|#{pane_current_command}'` + `split /\|/, $line, 3`.
The real-branch test's fake `tmux` was updated to emit the pipe-separated form
(matching real tmux behaviour) so the regression is locked in.

## Verification

- live: against real tmux in the lab, `tmux_pane_rows` now returns the pane and
  `resolve_claude_live_tmux_pane(<sid>)` resolves `%0` for the live
  `claude --resume <sid>` TUI; injecting a turn into that pane via
  `tmux send-keys` produced a real Claude reply (`7 times 8 is 56.`)
- Docker tests pass; `lib/Telegram/Claude/Manager.pm` stays at `100%`
